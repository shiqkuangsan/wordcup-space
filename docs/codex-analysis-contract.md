# Codex Analysis Contract

## 目标

Codex 可以使用本地数据、公开网页和 agent skills 辅助分析世界杯比赛，但分析本身不直接改变真实资金。世界杯期间的主线任务是每场比赛的赛前比分预测，下注只是从比分预测、盘口和风控派生出来的执行决策。任何下注动作都必须通过本地 API 的 `dryRun` 预览和用户确认。

## 输入顺序

分析一场比赛时按这个顺序取数：

1. 本地 `matches`：比赛、开球时间、场馆、阶段和来源。
2. 本地 `odds_snapshots`：你实际看到的盘口和赔率。
3. 本地 `portfolios` / `risk_profiles`：可用额度和风控上限。
4. `worldcup2026` provider：赛程、场馆、比分状态和积分榜补充。
5. `sports-skills@football-data`：球队、赛程、近期状态和公开比赛上下文。
6. `sports-skills@betting`：赔率转换、去水、EV、Kelly、串关计算。
7. `sports-skills@markets` 或 `world-cup` skill：预测市场和市场变化信号；只作为参考。

如果外部工具不可用，继续使用本地数据和手工/网页来源，并把 `dataQuality` 降级。

## 比分预测主线

每场比赛先处理 `codex_predictions`，再讨论 `bet_intents`：

1. 先确认比赛未开赛，且在当前可预测窗口内。
2. 先形成概率底座：预期进球、胜/平/负概率、Top score distribution、主预测方向和主预测比分。
3. 主预测比分必须和主预测方向一致；若单一最高概率比分与主方向不一致，只能作为比分分布参考，不能包装成第二套主预测。
4. 再给出或复核全场比分预测，例如 `加拿大 2-1 波黑`。
5. 若新信息改变判断，开赛前更新同一条预测记录；不要新增重复预测。
6. 下注决策必须引用当前比分预测，并说明该盘口如何从比分路径派生。
7. 比赛结束后回填实际比分，核心统计以 exact score 命中率为主；胜平负命中只作为派生参考。

预测可以没有下注；下注不能替代预测。即使 Codex 最终选择不下注，也要保留“预测 / 暂缓 / 不预测”的决策痕迹，避免赛后补预测。

系统内置的比分概率底座位于 `src/domain/prediction-model.ts`，吸收了 Elo/评分差转预期进球、Poisson 比分矩阵、单场波动修正和概率收缩这些优点。它只接受本地或明确来源的评分/修正输入，不读取外部 skill 的 `data/live` 快照，也不把样例 Elo 当作权威实时评级。写入 `codex_predictions` 时，可把模型输出作为 `modelSnapshot` 传给 `/api/predictions`，系统会把它和 sources 一起保存在 `sources_json`。

2026 世界杯由美国、加拿大、墨西哥联合举办。涉及这三支球队时，赛前预测必须显式评估东道主/准主场优势，包括观众、场地、旅行、气候适应和心理因素；但主场优势不能机械推导为主胜。

小组赛第二轮和第三轮必须先评估出线形势，再发布比分或下注建议。2026 赛制为 12 个小组每组前二直接晋级，另有 8 个成绩最好的小组第三晋级 32 强；因此当前积分、净胜球、进球数、末轮对手、第三名竞争和“保平/争胜/刷净胜球/避免淘汰”的动机会直接影响比分路径、让球盘和大小球。

## 自主下注证据门槛

Codex 自主下注前必须先通过证据门槛。未通过时只能输出 `wait` 或 `pass`，不能创建可执行下注决策。

Codex 自主下注必须先按“本周组合”分析，而不是逐场孤立分析。每次需要 Codex 决策时，先形成本周候选池和组合计划，再决定哪些比赛做单场、哪些比赛做串关腿、哪些放弃，以及总下注额和每日风险敞口。

同一场比赛可以产生多张 Codex 注单。系统不得默认“一场只下一单”：如果不同盘口各自通过证据门槛，Codex 可以在同一场同时买单关、把同一场纳入一张或多张串关，或单关和串关并存。每张票必须有独立理由、赔率、金额、失败条件，并把相关性计入当日风险。

用户给出的盘口想法默认是思路输入，不等于下单指令。Codex 必须独立评估该盘口的结算形态、价格、证据和组合风险；只有通过证据门槛并明确进入执行计划，才可以准备下注。

串关腿是独立决策池，不要求同时存在同盘口单场下注。允许“只进串关、不买单场”，也允许同一场比赛单场买 A、串关用 B；前提是每个 leg 都通过自身证据门槛，且组合后的相关性和总风险可接受。

串关可以是组合包，不限于一张。允许同一天设计多个 `2串1`、`3串1`、`4串1`，但要分清核心票、小额试探票和进攻票；每张串关不得超过串关单票风控，组合总额不得突破当日亏损上限，也不得全部押在同一个脆弱假设上。

每次串关执行包默认至少纳入一张比分串候选，除非 Betway 没有可验证的波胆价格。比分串可以是 `2串1` 或 `3串1`，用户可在单次执行中授权增加数量。比分串属于高波动进攻票，不替代核心单关或低波动串关；每条 leg 必须来自已发布的 Codex 比分预测，并核对当前 Betway 波胆赔率。

金额按资金比例和信心等级决定，不按固定面额决定。`10/20/30` 只是早期测试样本，不是 Codex 默认上限；当证据充分、赔率过线且风险敞口允许时，`50` 或更高金额可以是正常下注。默认风控参考：普通单场不超过 Codex 余额 10%，高信心单场不超过 20%，串关单票不超过 5%，单张串关最多 7 腿。每天 Codex 真实资金下注本金总额不得超过当天 Codex 资金的 25%，包含单关、串关、滚球、已准备待提交和已提交注单；该本金上限优先于单日最大亏损 40% 的宽口径约束。

亚洲盘的 0.25 / 0.75 线必须显式检查半赢/半输路径。若 Codex 看好的核心赛果只对应半赢，例如小 2/2.5 遇到正好 2 球只赢一半，或半场 -0.5/1 遇到一球领先只赢一半，不能把它当作强信心核心单关；应优先寻找更干净的盘口、降低金额、改作观察/串关腿，或直接放弃。

User 和 Codex 是完全隔离的决策主体。分析 Codex 自主下注时，不得因为 User 已经下注同一场、同一盘口、同一方向或同一金额而自动跳过。User 注单只能作为背景信息或对照样本，不计入 Codex 风险敞口。

必查项：

| 项 | 要求 |
|---|---|
| 当前盘口 | 必须来自 Betway 当前页、截图或已记录 odds snapshot，并确认赔率格式。 |
| 赔率格式 | 必须区分欧盘、港盘、马来盘；不确定时不得写入可执行赔率。 |
| 市场对照 | 至少对照一个独立盘口/市场来源，或明确说明没有可靠对照并降级为 `wait`。 |
| 小组形势 | 小组赛第二轮/第三轮必须核对当前积分、净胜球、末轮对手和第三名晋级路径。 |
| 近期状态 | 必须查看双方近期战绩或状态指标。 |
| 历史交锋 | 必须查看 head-to-head，并说明样本是否太老/太少。 |
| 阵容伤停 | 必须查看伤停、停赛、预计首发；未公布时明确标注不确定。 |
| 反方证据 | 必须写出不支持下注的主要证据。 |
| 执行条件 | 必须写下注金额、最低可接受赔率、放弃条件。 |

如果缺少当前盘口、近期状态、阵容伤停或市场对照中的任一项，默认 `recommendation=wait`。观察倾向不能升级为 `bet_intent`。

## 输出 JSON

Codex 在创建 `bet_intent` 前必须先形成这个结构：

```json
{
  "sources": [
    {
      "name": "local odds snapshot",
      "url": null,
      "capturedAt": "2026-06-10T12:00:00.000Z",
      "note": "Betway screenshot provided by user"
    }
  ],
  "dataQuality": "low",
  "market": "full_time:moneyline",
  "selection": "墨西哥胜",
  "modelProbability": 0.54,
  "fairOdds": 1.85,
  "marketImpliedProbability": 0.51,
  "expectedValue": 0.03,
  "riskTier": "normal",
  "confidence": "medium",
  "recommendation": "bet",
  "stake": 50,
  "minimumExecutableOdds": 1.9,
  "opposingEvidence": [
    "阵容未公布",
    "历史交锋样本过老"
  ],
  "abandonIf": "临场折合欧盘低于 1.90，或首发显示主力前锋缺阵。",
  "rationale": "简短说明概率、盘口、风险和主要不确定性。"
}
```

枚举约定：

| 字段 | 允许值 |
|---|---|
| `dataQuality` | `low`、`medium`、`high` |
| `recommendation` | `bet`、`pass`、`wait` |
| `riskTier` | `normal`、`high_confidence`、`longshot`、`parlay`、`parlay_aggressive` |
| `confidence` | `low`、`medium`、`high` |

## 写入规则

分析完成后，先调用：

```http
POST /api/intents
```

并传：

```json
{
  "dryRun": true
}
```

只有用户确认后，才去掉 `dryRun` 正式创建。Codex 不得因为外部工具给出正 EV 就直接创建真实下注或结算。

## 成交记录一致性

真实成交以平台成功页、确认号或用户明确同步为准。准备好的 Betway 票夹不是成交记录；如果用户没有最终提交、提交失败或页面出现错误，不得创建 `bet_slip`。

记录成交时必须核对：

| 项 | 要求 |
|---|---|
| 赛事 | 截图/页面中的球队、时间必须与 intent leg 匹配；非世界杯比赛只能用 `matchText` 文本记录。 |
| 盘口 | 盘口、选择和 line 必须与 intent leg 匹配；搜索结果里的 `先开球` 等子玩法不得误记为正赛盘口。 |
| 金额 | `stake` 取平台成功页的总投注额；串关不得把单关金额误加进去。 |
| 总赔率 | `bet_slips.final_odds` 记录整张注单的最终赔率；串关为总赔率。 |
| 单腿赔率 | `bet_slip_legs.final_odds` 记录每个 leg 的最终单腿赔率；不得把串关总赔率复制到所有 legs。 |
| 确认号 | `confirmationRef` 必须来自平台成功页；用于后续去重和核对。 |
| 截图 | 有截图时保存 `confirmationScreenshotPath`，便于赛后复盘。 |

`POST /api/placed-bets` 支持 `legs[]` 直录串关注单。串关写入后必须用注单号核对：父级 slip 的总赔率、总 stake、潜在返还正确，且每条 `bet_slip_legs.final_odds` 仍是单腿赔率。

## 解释规则

- 所有数据型判断必须有来源；没有来源只能写成假设。
- 预测市场价格不能直接等同真实胜率。
- `sports-skills@world-cup` 是可选 premium/read-only 能力；不可用时不阻塞本地分析。
- 自动比分同步不能自动结算注单；平台结算仍需要用户确认。
- 输出要说明主要反方证据，例如伤停不确定、阵容未公布、盘口移动过快或数据源过期。
