# 操作手册

## 目标

使用 Codex 分析每场世界杯比赛，辅助你判断喜欢的球队是否值得买，并记录 User 和 Codex 两套决策来源。真实平台账户可以只有一个；系统内部通过决策来源、执行人、资金归属和真实资金标记来区分。

## 本地服务隔离

开发和使用必须隔离，避免 Codex 开发时影响用户正在看的系统。

| 服务 | 命令 | 端口 | 用途 |
|---|---|---:|---|
| 开发服务 | `pnpm dev` | `3107` | Codex 开发、调试、浏览器验证。 |
| 使用服务 | `pnpm start` | `3108` | 用户日常查看的本机稳定服务。 |
| 一键更新 | `pnpm run run` | `3108` | 先 build，再启动使用服务。 |

执行规则：

- 开发过程中允许运行 `pnpm lint`、`pnpm test`、`pnpm build` 验证代码。
- `pnpm build` 只更新构建产物，不等于更新用户正在使用的 `3108` 服务。
- 未经用户明确要求，不启动、停止或重启 `3108`。
- 用户确认更新后，才执行 `pnpm run run` 或用最新 build 重启 `3108`。

## 双主体记录

系统必须同时记录用户投入和 Codex 决策，并在统计时明确区分：

| 主体 | decision_by | 用途 |
|---|---|---|
| 用户 | `user` | 记录你自己的判断、投入、个人复盘。 |
| Codex | `codex` | 记录 Codex 的模型分析、自主决策、额度使用和复盘。 |

关键原则：

- 所有预测、持仓、台账和复盘都要记录 `decision_by`。
- 所有实际下单记录都要记录 `placed_by`；默认都是 `user`，表示你代为执行。
- 真实平台账户可以只有一个，用 `platform_account` 记录，默认使用 `betway-main`。
- 涉及资金的记录必须归属到明确的 `portfolio_id`，例如 `user` 或 `codex`。
- 每张注单都要记录 `is_real_money`，用于区分真实资金和模拟记录。
- 同一场比赛可以同时存在用户决策和 Codex 决策，不能互相覆盖。
- User 和 Codex 完全隔离：User 已买某场、某盘口、某金额，不能作为 Codex 跳过该比赛的理由。
- Codex 可以买和 User 完全相同的比赛、盘口、方向和金额，也可以反向买；只看 Codex 自己的证据门槛、风控和预算。
- dashboard 默认同时展示两套结果，也要支持单独过滤用户或 Codex。
- Codex 可以参考用户提供的数据，但必须标记数据来源；用户自己的投入不能混入 Codex 的额度统计。
- Codex 的真实资金单，本质是 `decision_by=codex`、`placed_by=user`、`is_real_money=true`。

## 聊天代操作优先

高频使用时，用户不需要进入页面手动填表。默认流程是：用户在当前对话里发截图或口述，Codex 解析成结构化 payload，先 dry-run，展示将写入的注单、legs、余额变化和警告；用户确认后，Codex 再执行写入并回读核对。

页面主要用于查看、复盘、筛选和纠错；真实输入入口优先是聊天窗口加本地脚本/API。

| 输入方式 | 系统动作 |
|---|---|
| 注单成功截图 | Codex 调用 `pnpm record:placed-bet` dry-run；确认后 `--write`。 |
| 口述已下注 | Codex 追问缺失字段后同样走 placed-bet dry-run。 |
| 页面表单 | 作为备用入口，不是默认要求。 |
| 批量修正 | Codex 先列出影响范围，得到确认后再执行。 |

## 单场比赛流程

每场比赛按下面状态推进：

| 状态 | 含义 |
|---|---|
| `scheduled` | 已有赛程，但还没有正式分析。 |
| `priced` | 至少记录了一组博彩公司赔率快照。 |
| `analyzed` | 已记录模型概率和定性分析。 |
| `decided` | 已做决策：下注、放弃、观察，或作为串关一腿。 |
| `settled` | 已记录赛果和资金影响。 |
| `reviewed` | 已完成赛后决策质量复盘。 |

## 预测流程

第一版默认流程：

1. 把博彩公司赔率换算成隐含概率，并去除水位/抽水影响。
2. 用最优可得赔率建立市场基准概率。
3. 根据球队实力、近期状态、伤停、首发、场地/旅行/休息、战术对位、赛事背景做调整。
4. 对支持的市场输出概率：`1X2`、让球/亚洲盘口、大小球；数据质量足够时再考虑部分特殊玩法。
5. 比较模型概率和当前盘口赔率。
6. 按资金规则决定下注金额。
7. 开赛前记录决策。
8. 赛后记录结果、收盘线价值和复盘。

## Codex 自主下注纪律

Codex 可以自主选择每周下注哪些比赛、买什么盘口、是否串关和下注金额，但必须把“观察倾向”和“可执行下注”分开。

Codex 的基本工作单元是“本周下注组合”，不是单场孤立判断。每周先从赛程和当前盘口建立完整候选池，再统一决定：

- 哪些比赛适合单场。
- 哪些比赛只适合作为串关腿。
- 哪些比赛只观察或直接放弃。
- 本周是否需要买单场、买串关、两者都买，或者全部放弃。
- 每张单的金额、最低执行赔率、执行窗口和当日最大亏损约束。

创建一条 intent 不代表本周扫描结束。未入选的比赛仍要留在滚动观察池，尤其是后半周、尚未抓到 Betway 当前盘口、或需要临场阵容确认的比赛；只有在证据不足、赔率不达标或风险不合适时，才标记为放弃。

当本周下注分析 goal 处于激活状态时，Codex 要主动推进候选池扫描、盘口复核、组合排序和 intent 创建，不等用户逐场触发。用户的新消息可以调整优先级，但不是每场比赛分析的前置条件。

User 注单只能作为背景信息，不能替代 Codex 自主判断。Codex 的候选池必须从赛程和盘口出发，不因为 User 已经买过而移除。

用户给盘口建议时，默认只是提供思路，不是命令 Codex 必买。Codex 仍要自己判断：这个玩法是否适合当前比分路径、是否有足够赔率、是否存在更干净的替代盘口、是否挤占当天 25% 本金上限。

| 状态 | 含义 | 是否能下单 |
|---|---|---|
| 观察 | 有初步想法，但证据不足或盘口未确认。 | 否 |
| 放弃 | 证据足够，但没有价值或风险不合适。 | 否 |
| 决策 | 证据门槛通过，已给出盘口、金额和执行条件。 | 可以，成交后再记录注单 |

Codex 自主下注的最低证据门槛：

- 当前 Betway 盘口和赔率格式。
- 至少一个独立市场/盘口来源作对照。
- 双方近期战绩或状态。
- 历史交锋，并说明样本是否太老或太少。
- 阵容、伤停、停赛或预计首发；未确认时必须写成不确定风险。
- 选择该盘口的原因，例如为什么买大小球而不是胜平负。
- 反方证据。
- 下注金额、最低可接受赔率和放弃条件。

缺少当前盘口、近期状态、阵容伤停或市场对照时，默认只能 `wait`，不能创建可执行 intent。

### 2026-06-11 实战教训

韩国 vs 捷克首个 Codex 候选曾过早从“观察倾向”升级为“待执行决策”。问题是只基于 Betway 可见盘口和少量外部观点，未充分检查历史交锋、近期战绩、阵容伤停和市场总进球定价。复核后该 intent 已取消。

以后遇到类似情况，必须先降级为观察，补齐证据后再决定是否下注。

同日另一条规则修正：墨西哥 vs 南非不能因为已有 User 注单而从 Codex 候选池移除。User/Codex 是两个独立账本和决策主体，同场同向同金额都允许。

同日再修正一条流程规则：Codex 不能只围绕用户刚提到的一场比赛做逐场决策。自主下注时必须先看本周所有可买比赛，按组合管理思路统一决定单场、串关、放弃和资金分配。

同日补充：本周创建了某个单场 intent 后，不能默认其它比赛“一律不看”。其它比赛继续进入滚动观察池，按盘口更新、伤停更新和开赛时间逐步复核。

同日执行边界确认：Codex 可以自己操作 Chrome 准备下注，也可以由用户代下，但任何真实下注的最终提交按钮必须停下来等用户确认。只有提交成功后，系统才记录 `bet_slip`。

2026-06-14 实操修正：Betway 真实连续下单优先由用户手动执行。Codex 负责分析、盘口清单、金额、风控和系统记录；Chrome 主要用于看盘、核对盘口、辅助定位。原因是 Betway 票夹会残留旧票、成功回执、推荐组合和自动填金额，连续自动点击容易把“清理票夹”变成真实提交。若页面出现混合状态，Codex 应停止操作，让用户手动下单并用成功截图回传。

2026-06-14 滚球对冲经验：澳大利亚 2-0 土耳其时，已有 `User 2-0 波胆` 和 `Codex 澳大利亚/土耳其小 2.5 串关` 两张票受益于比分不再变化。用户买入 `大 2.5` 港盘 `1.38`、35 元作为对冲。计算口径：

- 若维持 2-0：原票盈利约 `149.60 + 53.25`，对冲亏 `35`，组合仍有较大正收益。
- 若再进 1 球或更多：原票亏 `20 + 25`，对冲盈利 `35 * 1.38 = 48.30`，组合约 `+3.30`。
- 对冲保本线约为 `45 / 1.38 = 32.61`，实际 35 元略高于保本线，属于合理防守金额：牺牲一部分上行收益，换取逆向分支基本不亏。

以后评估滚球对冲时，先列出“原路径继续命中”和“逆向路径发生”两个分支，使用港盘盈利额计算保本下注额，再判断是否接受牺牲的上行空间。若同时涉及 User 和 Codex 账本，必须明确是在评估单一账本还是平台实战组合。

同日反买教训：澳大利亚 2-0 土耳其说明，赛前分析主线可能被临场走势打穿。以后不能把赛前判断当执念；当被低估一方持续压迫、主场/半主场氛围明显、热门方迟迟打不开、赔率漂移或比赛节奏明显反向时，允许建立小额反向仓位。反买不是赌气翻桌，而是买“原判断失败路径”：金额要小、触发条件要清楚、不能覆盖或破坏整体风控。

### 每日四场比赛的标准流程

以后用户每天通常只会给当天比赛列表，Codex 必须自主完成完整循环：

| 步骤 | Codex 要做什么 | 输出 |
|---|---|---|
| 赛程确认 | 确认当天比赛、开球时间、是否已经开赛。 | 今日候选池 |
| 比分预测 | 先更新每场比分预测；预测和下注分开记录。 | 每场预测比分和信心 |
| 盘口扫描 | 在 Betway 查看常用盘口：胜平负、让球、大小球、半场、双方进球、球队进球、波胆、串关可用性。 | 可下注市场表 |
| 单关计划 | 自主决定买哪些、放弃哪些、观望哪些；一场可以多张，也可以零张。 | 单关清单、金额、最低赔率 |
| 串关计划 | 单独构建串关池，不要求 leg 必须也是单关；普通串关和比分串分开。 | 串关清单、金额、legs |
| 风控压缩 | 当天所有真实资金本金不得超过 Codex 当天资金 25%。超出时先砍高波动票和半赢盘。 | 最终执行包 |
| 执行准备 | 优先由用户按清单手动下单；Codex 仅在页面干净且单张票明确时用 Chrome 辅助准备，停在 `请下注` / `确认下注` 前。 | 用户最终确认 |
| 成交记录 | 用户提交成功并给截图/编号后再写系统。 | `bet_slip`、legs、账本流水 |

### 盘口结算形态纪律

亚洲盘 0.25 / 0.75 线要先看“命中后到底赢多少”：

- 小 `2/2.5`：正好 2 球只赢一半。
- 让 `-0.5/1`：只赢 1 球只赢一半。
- 让 `-1`：赢 1 球走水，赢 2 球才全赢。
- 大小 `2.5`、让 `-0.5`、胜平负等更接近“结果命中就完整结算”的表达。

半赢盘不是禁止，但不能自动当核心强信心单关。若看好的主路径只给半赢，应优先寻找更干净盘口、降低金额、改作串关小腿，或放弃。

## 资金规则

Codex 可配置独立额度。初始建议额度约 `1000`，以你实际分配为准。

| 规则 | 默认值 |
|---|---:|
| 普通单场下注上限 | 资金的 `10%` |
| 高信心单场下注上限 | 资金的 `20%` |
| 串关下注上限 | 资金的 `5%` |
| 每日真实资金本金上限 | 资金的 `25%` |
| 单日亏损上限 | 资金的 `40%` |
| 出手最低优势 | `3%` 期望值 |
| 自主下注最低数据质量 | `medium` |

不追损。连续亏损时降低风险，而不是加大下注。

## 决策模式

| 模式 | 目的 | 规则 |
|---|---|---|
| `support` | 帮你判断是否支持喜欢的球队 | 重点说明风险、公允赔率、情绪溢价和替代选择。 |
| `autonomous` | Codex 自主选择模拟下注 | 必须遵守资金规则并记录理由。 |
| `parlay` | 模拟串关 | 只允许小额，并记录相关性风险。 |
| `pass` | 不下注 | 放弃也是有效决策，也要计入统计。 |

## 资金账本

| 账本 | 归属 | 说明 |
|---|---|---|
| `user` | 用户 | 记录你的判断、投入、真实/模拟决策。 |
| `codex` | Codex | 记录 Codex 的自主决策和额度使用，可包含真实资金或模拟记录。 |

用户账本和 Codex 账本分别计算余额、盈亏、ROI、回撤和风险敞口。跨账本只做对比，不做合并收益。

如果某条记录只是“用户提供盘口给 Codex 参考”，它属于数据来源，不自动算作用户下注。

如果你用同一个平台账户代 Codex 下单，记录方式是：

```text
platform_account = betway-main
portfolio_id = codex
decision_by = codex
placed_by = user
is_real_money = true
```

## 可视化目标

- 资金曲线和回撤。
- ROI、yield、命中率、平均赔率。
- User vs Codex 的收益、命中率、赔率质量、收盘线价值对比。
- 按球队、小组、市场、博彩公司统计风险敞口。
- 模型校准：预测概率区间 vs 实际命中率。
- 按市场和博彩公司统计收盘线价值。
- 单场比赛卡片：赛前理由和赛后复盘。
- 串关树和相关性说明。

## AI 使用规则

AI 可以总结新闻、比较球队、审查推理、起草比赛笔记。不能编造数据。每个数据型判断都要引用已记录来源，否则标记为假设。

Codex 不需要把强 AI 能力内置到 app。默认策略是：

- app 负责可靠记录：赛程、盘口、决策、执行、注单、结算和资金账本。
- Codex 负责研究和解释：必要时调用 agent skills、公开网页或本地 provider 数据，再把结果整理成结构化理由。
- `machina-sports/sports-skills` 只作为 Codex/agent 工具安装，不作为 Next.js 运行时依赖。
- `rezarahiminia/worldcup2026` 只作为赛程/球队/场馆/比分 provider，不直接生成下注建议。
- 所有 Codex 决策默认先走 `dryRun` 或人工确认；外部工具不得绕过 `/api/intents`、`/api/bet-slips` 和风控规则。
- 市场价格、预测市场和新闻情绪只能作为参考信号，不能写成确定性结论。

## 当前支持玩法

- 全场/半场：胜平负、让球、大小球。
- 串关：最多 7 legs，默认小额风控。
- 特殊玩法：第 N 个进球是哪只队伍。
- 结算支持：赢、输、走水、半赢、半输、提前兑现、取消/无效。

默认资金都是真实资金；只有明确选择“模拟记录”或 API 传 `isRealMoney=false` 才作为模拟。
